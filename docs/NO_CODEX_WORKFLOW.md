# No-Codex Workflow for Capital Radar

## Purpose

This runbook defines how to keep improving Capital Radar when Codex quota is unavailable.

The goal is to avoid broad, irreversible edits. Work proceeds through documentation, issue control, narrow GitHub changes, and verification-ready mission packets.

## Core principle

When Codex is unavailable, do not attempt large code rewrites manually.

Instead:

1. Stabilize doctrine.
2. Write exact mission contracts.
3. Create narrow branches.
4. Edit one file or one module at a time.
5. Use existing scripts for verification when possible.
6. Archive every decision.

## What can be done safely without Codex

### Safe

- Create issues.
- Create doctrine docs.
- Update README.
- Update roadmap.
- Write mission packets.
- Audit existing files.
- Modify manifest policy text.
- Make small static-copy changes.
- Add validation intent before implementation.

### Medium risk

- Edit a renderer if the change is small and localized.
- Edit CSS if the selector is clearly scoped.
- Modify a manifest section only if the destination and validation are clear.
- Update one generator if its input/output contract is known.

### High risk

- Build-pipeline rewrites.
- Registry migration.
- Multi-section layout changes.
- Generated output edits by hand.
- Removing validators.
- Deployment-trigger changes.
- Large CSS refactors.

Avoid high-risk work until Codex is available or a local dev environment is used.

## Manual work sequence

### Step 1 — Control map

Create or update a GitHub issue that states:

- objective
- current problem
- desired state
- files likely involved
- acceptance criteria
- verification command

### Step 2 — Doctrine check

Before implementation, confirm the task matches:

- `AGENTS.md`
- `docs/CAPITAL_RADAR_KERNEL.md`
- `docs/OPENCLAW_MISSION_TYPES.md`
- `docs/MODULE_REINTEGRATION_MAP.md`

If the task contradicts doctrine, update doctrine first or stop.

### Step 3 — Branch

Create a narrow branch:

```text
mission/<short-task-name>
```

Examples:

```text
mission/compress-cross-asset-into-macro
mission/update-readme
mission/macro-cycle-badge
```

### Step 4 — One-file or one-module edit

Change as little as possible.

Preferred order:

1. documentation
2. manifest policy or description
3. renderer copy / small layout
4. scoped CSS
5. generator output contract
6. validator

### Step 5 — Verification plan

Use the narrowest command:

```bash
npm run build:fast
```

for UI-only changes.

Use:

```bash
npm run validate:lane
```

for validation-only changes.

Use:

```bash
npm run build:prod
```

only for production-level verification.

If verification cannot be run from the current environment, say so explicitly and create a checklist for the next Codex/local run.

### Step 6 — Archive

Update the controlling issue or a roadmap note with:

- what changed
- why it changed
- verification status
- what remains

## Recommended sequence while Codex is unavailable

### Mission 1 — Doctrine foundation

Files:

- `AGENTS.md`
- `docs/CAPITAL_RADAR_KERNEL.md`
- `docs/OPENCLAW_MISSION_TYPES.md`
- `docs/MODULE_REINTEGRATION_MAP.md`
- `docs/NO_CODEX_WORKFLOW.md`

Goal:

Give future OpenClaw/Codex work a stable rule base.

Verification:

- no build needed
- check for internal consistency

### Mission 2 — README correction

Files:

- `README.md`

Goal:

Replace stale Financial Report/Morning Brief language with current Capital Radar identity, build commands, workflow, and homepage architecture.

Verification:

- documentation review

### Mission 3 — Cross-Asset Lens compression spec

Files:

- `docs/MODULE_REINTEGRATION_MAP.md`
- optional issue comment

Goal:

Define exactly how Cross-Asset Lens becomes a Macro confirmation row before touching UI code.

Verification:

- doctrine review

### Mission 4 — Macro renderer implementation

Files likely involved:

- `components/radar/decision-brief/render.cjs`
- related Macro CSS or renderer style export
- possibly state reads already generated by `scripts/generate-market-lens-state.cjs`

Goal:

Render Cross-Asset Lens as compact Macro evidence, not standalone section.

Verification:

```bash
npm run build:fast
```

### Mission 5 — Market Tape compression

Same pattern as Mission 4, but only after Cross-Asset Lens is stable.

## Stop conditions

Stop and wait for Codex/local verification if:

- a change requires multiple scripts to be rewritten
- a build result is unknown
- generated artifacts conflict with manifest policy
- a validator fails and the reason is unclear
- the UI requires visual QA that cannot be inspected
