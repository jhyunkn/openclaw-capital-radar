# Capital Radar Architecture

## Current framework

Capital Radar is currently a static HTML application generated by Node scripts. It is not currently a Next.js, Vite, or conventional React app.

The production build flow is script-driven:

1. Node scripts generate JSON artifacts into `outputs/` and related folders.
2. Node injectors render sections into `index.html`.
3. `scripts/build-vercel.cjs` copies static output into `public/` for Vercel.
4. Vercel serves the `public/` directory.

The app should be treated as a Node-rendered static intelligence interface until a deliberate framework migration happens.

## Target organization

The repository now reserves explicit workspaces for a component-oriented future:

```txt
/app
  Future route/page workspace.
/components
  /layout
  /radar
  /ui
/data
  /mock
/lib
  /scoring
  /types
  /utils
/prototypes
  /html
  /archive
/docs
```

## Production principle

Production routes and generated pages must not directly paste standalone HTML experiments. Standalone HTML is a prototype layer only.

Useful prototype ideas should be extracted into:

- a data shape in `lib/types`
- mock JSON in `data/mock`
- scoring or interpretation logic in `lib/scoring`
- reusable visual components or component specs in `components/radar`
- production renderers only after mock-data validation

## Existing production system

Existing production code remains script-rendered for build stability. Large-scale framework migration is intentionally not part of this hygiene pass.

Existing folders such as `scripts`, `assets`, `outputs`, `public`, `pages`, and root `index.html` remain in place because they are part of the working build and deployment path.

## Structural decision record

- Created `app/` as a future route workspace rather than moving production code into it prematurely.
- Created `components/` to separate layout, generic UI primitives, and radar-specific visual intelligence components.
- Created `data/mock/` to prevent hardcoded one-off visual values from becoming the component contract.
- Created `lib/types/` and `lib/scoring/` so decision visuals can be driven by explicit schemas and reusable scoring logic.
- Created `prototypes/html/` and `prototypes/archive/` so visual experiments have a home without polluting production routes.
- Preserved existing static pipeline to avoid unnecessary build risk.

## Migration rule

A future move to Next.js, Vite, or React should happen only after the component contracts stabilize. Until then, Capital Radar should keep using the current static pipeline while progressively extracting reusable radar components.
